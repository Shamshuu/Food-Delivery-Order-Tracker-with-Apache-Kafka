1. Project Overview & Purpose
This project is a Real-Time Food Delivery Order Tracker built using an Event-Driven Architecture (EDA).

What It Does:
Simulates customers placing orders at restaurants, which progress through a five-stage lifecycle: PLACED → CONFIRMED → PREPARING → OUT_FOR_DELIVERY → DELIVERED.
Streams these status transitions in real-time to a central message broker.
Uses two separate backend consumers to independently capture, process, and act on these events (one to track active order status, and another to perform live business analytics).
Visualizes the live progress of all active orders on a modern, dark-themed browser dashboard.


Purpose (The "Why"):

The core purpose is to demonstrate Decoupling and Scalability in microservice architectures:

Loose Coupling: 
The order service (producer) has zero knowledge of the services reading its data. We can add new services (e.g., a fraud detector or email notifier) at any time without modifying the producer.

Fault Tolerance: 
If one consumer crashes, it does not affect other consumers. Once the crashed consumer is restarted, it resumes reading from exactly where it left off, ensuring no data loss.

Strict Sequencing: 
Ensures that status transitions are processed in the correct order (e.g., you never see DELIVERED before PLACED).


2. How the Technologies Help

Docker & Docker Compose: 
Package ZooKeeper, Kafka, and Kafdrop into containers. This allows the system to run on any OS with a single command (docker compose up), avoiding complex local installations.

Apache Kafka: 
The event streaming broker. It acts as an immutable, append-only log. It organizes messages into topics (like order-events) and divides them into partitions (allowing parallel consumption), serving as a reliable backbone.

ZooKeeper: 
Coordinates the Kafka broker metadata, handles leader election for partitions, and keeps track of active nodes.

Kafdrop: 
A visual web UI that exposes the state of Kafka. It enables developers to debug the system by inspecting partition layout, reading payloads, and observing active consumer groups without using CLI commands.

Node.js & kafkajs: 
Node's asynchronous, event-driven runtime is highly suited for streaming systems. It allows us to handle incoming Kafka messages and run an HTTP API concurrently in the same thread.

Express.js: 
Used in Consumer A to serve a fast, lightweight API endpoint (GET /state) with CORS enabled so the frontend can query the current status.

Vanilla CSS & JS: 
Serves the frontend board with zero build tools (like Webpack or Vite), loading instantly. Standard Vanilla CSS is customized with CSS variables, gradients, blurs, and keyframe transitions to create a premium, glassmorphic dark-mode dashboard.


3. Detailed Functionality & Implementation
A. Event Generation: producer.js

Staggered Concurrency: Simulates multiple active orders in parallel. It selects a random customer, restaurant, and menu items from fixtures.json, staggers the start times of the orders, and updates their states with random intervals (2–8 seconds).

Partition Routing (Message Key): Crucially, it sets the message key to order_id (e.g. ORD-001). Kafka hashes this key to ensure all status updates for ORD-001 land in the same partition, guaranteeing they are read sequentially.

Resiliency: Implements an explicit retry block that intercepts broker connection hiccups and attempts to republish the event up to 3 times with a delay.


B. Active Order Tracking: consumer_a.js

State Management: Listens under the status-tracker consumer group. It maintains an in-memory dictionary activeOrders.

Log Transitions: Logs updates (e.g., [UPDATE] ORD-001: PREPARING -> OUT_FOR_DELIVERY).

Order Completion: When an order transitions to DELIVERED, Consumer A prints a completion summary (e.g., [COMPLETE] ORD-001 | Pizza Place | ~15 min.) and deletes the order from activeOrders so the dashboard stops tracking it.

State Persistence: Writes activeOrders to state.json every 10 seconds. On restart, it automatically loads this file to resume tracking without losing historical states.

State API: Hosts the Express server running on port 5000 to share activeOrders with the frontend.


C. Live Analytics: consumer_b.js

Independent Consumption: Listens under the analytics consumer group. It receives a copy of the exact same message stream as Consumer A, demonstrating Kafka's publish-subscribe flexibility.

Aggregation: Tallies metrics in two simple dictionary structures: messagesPerRestaurant (counts total orders handled per restaurant) and statusCounts (counts occurrences of each status type).

Console Snapshots: Automatically prints a formatted table of current restaurant and status frequencies to standard output every 15 seconds.


D. The Frontend Board: frontend/
Dynamic Polling: app.js polls http://localhost:5000/state every 2 seconds, displaying the connection state.

Order Progress Bars: Each order card renders a 5-step horizontal tracker (PLACED, CONFIRMED, PREPARING, OUT_FOR_DELIVERY, DELIVERED). Completed steps light up in distinct colors depending on the status.

Smart Completion (UX Polish): Since Consumer A deletes orders immediately upon delivery, they disappear from the API response. To avoid cards vanishing abruptly, the frontend detects if an active order has gone missing, manually forces its progress bar to 100% (DELIVERED), holds it on screen for 4 seconds, and then triggers a CSS fade-out transition before removing it from the DOM.
